# Agentic SDLC: Design

This document is the design summary of the starter. The authority for every decision is the specification document "Agentic SDLC Specification and Architecture"; section references below point there. Where the starter simplifies, the simplification is named in "What this starter defers".

## Definitions

- The **model** is the reasoning engine, the large language model. The **harness** is everything wrapped around the model so the model can finish a job: instructions, tools, orchestration logic, guardrails, observability. An **agent** is a model plus a harness running a loop; Claude Code is the agent, and the harness is roughly 90 per cent of what determines behaviour. Everything in this repository is harness.
- The **main agent** is the top-level Claude Code the developer talks to; it plays the concierge role at intake and the orchestrator role across the run. A **subagent** is the same Claude Code, spawned by the main agent, specialised by an agent definition under `.claude/agents/`. An agent definition is harness, not an agent.
- An **agent role** is a named unit of agentic behaviour with a declared contract (the input artefacts it reads plus the output artefacts it writes) and a recommended model strength. An agent role is realised at run time as a subagent or as a skill; the manifest's `primitive` field records which.
- An **initiative** is the body of work one accepted intent sets in motion, tracked under one initiative identifier for its whole life. A **run** is one pass of the pipeline for one intent, from classification to the closing of the run record.
- A **recipe** is a named, reusable stage sequence for one target at one magnitude; the **execution plan** is the per-run plan the orchestrator composes by pruning a recipe.

## The seven axioms (specification section 1.4)

1. **Artefact bus.** Work moves between agents only as durable artefacts on the artefact tree; spawning is control flow, not work transfer, and no downstream agent depends on a completion summary.
2. **Contract.** Every agent role declares its inputs and outputs; the registry is generated by scanning the declarations.
3. **Semantic signals.** A declared input is information that must be present, not a format that must be matched: missing information is elicited, inferred, or defaulted, and an absent optional input triggers a named degraded path written in the agent's file.
4. **Single caller.** The orchestrator is the only caller of subagents; Claude Code permits two spawn layers, so all sequencing, gating, and consolidation live in the one main agent, and parallelism is a fan-out in one message.
5. **Same agent, two bindings.** A standalone run and a composed run use the same agent definition; only the source of each input differs.
6. **Recipe guidance.** The orchestrator is guided by recipes and is itself a model: a recipe-guided run is semi-hard, repeatable but not deterministic; deterministic enforcement is a separate axis (hooks and permission policy), largely deferred in this starter.
7. **No new technology.** Everything runs on Claude Code, markdown, YAML, and Python run with uv.

## The shared memory (specification chapter 2)

- `sdlc.config.yaml` at the target repository root is the single per-project profile: project facts, artefact-tree root, product locations per generation target, validation commands, review roster and three-tier severity model, the failure-pattern catalogue pointer, the schema profile, and the agent availability switch. The harness installs verbatim; only this file changes per project. Schema: `.claude/config/sdlc.config.schema.yaml`; validator: the Pydantic model in `.claude/scripts/scaffold.py`.
- The artefact tree (default `ai_docs/`) has two tiers. Tier one, the project spine, is singular and updated in place: `charter.md`, `prd.md`, `architecture.md`, `reference/` (CONTEXT.md, testing conventions, ADRs, the operator reference, the schema snapshot), and the generated `agent-registry.md`. Tier two is one workspace per initiative under `initiatives/<initiative-id>/`: implementation plan, per-task specs with the task registry, explorations, task briefs, reviews, walkthroughs, reconciliations, and the run record. `initiatives/index.md` is the initiative registry, written only by the orchestrator, carrying a status sentence per initiative and the focus note.
- Artefacts are addressed by the composite join key `<initiative-id>/TASK-XXX`, so a consumer resolves any input by constructing a path, and idempotency needs no machinery: a file that exists at the constructed path and validates means the producing stage is skipped. The developer's regenerate override runs the producing stage anyway, overwriting in place while the provenance header appends a new modified entry.
- Every artefact conforms to a template under `.claude/templates/` and carries the append-only provenance header (created, modified, commits, agents, run-ids, back-refs, forward-refs). Templates are guidance, not hard gates: output is parsed tolerantly, and a missing section is flagged rather than rejected.
- Generated product artefacts (dbt models, ADP Foundry YAML, ad-hoc code) land in the application repository at the `product_locations` recorded in `sdlc.config.yaml`, never inside the artefact tree.
- The run record (`run-record/`, template `.claude/templates/run-record.md`) logs the detected intent, the composed plan with a one-line rationale per inclusion and exclusion, and the outcome per stage (ran, reused, failed).

## The agent model (specification chapter 3)

- Every agent role's file carries the four native frontmatter keys plus the extended manifest as additional frontmatter keys the harness reads: `inputs` (each a semantic signal with its source and resolution), `outputs`, `preconditions`, `intents`, `scope` (core, data-engineering profile, or bespoke), `model_floor`, `cost_tier`, `standalone`, `idempotency`, `primitive`, `phase`. `.claude/scripts/generate_agent_registry.py` scans both `.claude/agents/` and `.claude/skills/` and materialises `ai_docs/agent-registry.md`.
- Every subagent-realised role ends with the fixed four-section completion summary (Verdict, Escalations, Risks and inconsistencies, the read flag), an extract of its artefact; a question with material impact is never settled by assumption. Skill-realised roles return no summary: the main agent performed the work, and the gate is the confirmation inside the consultation.
- The tool grant is contract: a reviewer holds Write solely for its own review artefact and is denied Edit; a generator holds Write and Edit for the product. In this starter the fence is stated in each fenced file; the enforcing pre-tool-use hook is deferred.
- Model tiering: no role runs on the weak tier; the specification roles (requirements-navigator, arch-blueprint, implementation-planner, task-preparer) and the review-consolidator never drop below strong; the producing roles run mid at the base dial and strong at the full dial. The dial is set once at classification from the magnitude (`.claude/config/intents.yaml`), and the orchestrator passes the resolved model when it spawns each subagent.

## Composition (specification chapter 4)

- The concierge classifies intent on two dimensions: the target (eight, a closed enumeration in `.claude/config/intents.yaml`) and, for the three generation targets, the magnitude (bug-fix, feature-update, new-feature, new-capability, new-project, defined by what the intent does to the product).
- The recipe library lives under `.claude/config/recipes/`: one family per generation target (five recipes each, keyed by magnitude, with per-stage inclusion conditions) and one short recipe per non-generation target. One pipeline converges through the shared upstream (charter, requirements, architecture, plan, per-task spec), diverges at exactly one generation stage (ADP Foundry YAML, dbt model, or ad-hoc code), and converges through the shared downstream (risk classification, the reviewer subset, consolidation, walkthrough, reconciliation, the run record).
- The orchestrator prunes the selected recipe by four criteria: magnitude (inclusion conditions checked against the actual intent), idempotency, risk (the reviewer subset the risk-classifier recommends), and the availability switch. Three stages are mandatory in every composition that changes the product: review with consolidation, reconciliation, and the run record.
- Two composition primitives. A unit of work is a skill when the work is a conversation the developer drives (the four gated consultations, the ad-hoc code implement loop, access provisioning); it is a subagent when the work produces an artefact autonomously from declared inputs (everything else). The artefact is the next stage's prompt: the per-task spec is at once a durable artefact and the prompt the implementation executes.

## Entry and orchestration (specification chapter 5)

- `CLAUDE.md` stays thin (the two roles, the always-on rules, the loading rule); three harness-core skills carry the procedures: `intake` (session-start protocol, classification, elicitation flag, thresholds, edge cases, handover), `orchestration` (composition, the run loop, escalation triage, standing duties), and `prime` (project grounding plus the rebuild of one confirmed initiative). Harness-core skills are not catalogue entries and are excluded from the registry scan.
- Session continuity does not rely on compaction: the SessionStart hook (`.claude/hooks/build_initiative_catalogue.py`) injects the initiative catalogue and the reconstitution protocol; the developer confirms one initiative; the prime skill reloads the run record and the initiative's artefacts inline. After each consultation, the orchestrator proposes `/clear`; the rebuild is lossless where compaction is lossy.
- Handoff patterns: Pattern A, orchestrator-driven with gates, is the default; Pattern B invokes one agent role at a time by hand, rehydrating from the artefact bus; Pattern C, the programmatic issue-in pull-request-out run, is Phase 2.

## Cross-cutting (specification chapter 6)

- Grounding pulls external truth into artefacts: the warehouse-schema snapshot (schema-retrieval-warehouse), the ADP Foundry operator reference (foundry-operator-reference), and the living reference documents (context-agent, their single writer). A generator reads the snapshot, never the live source; a bespoke snapshot under `ai_docs/data-engineering/` is a stated exception, never a silent norm.
- The capped testing loop closes every generation stage: run the spec's Validation Commands, parse failures, fix, re-run; stop on zero failures, the three-iteration cap, or zero progress.
- The review panel is ten fresh subagents, one dimension each, spawned in parallel in one message, scoped by the risk-classifier; the review-consolidator de-duplicates, re-validates every finding against the code, ranks by severity, and emits one machine-readable verdict; the gate blocks on blockers only. Evidence discipline: no finding without a file and line trace.
- Reconciliation keeps the recorded plan agreeing with the implemented reality, with the code as ground truth and a write boundary drawn by ownership: overwrite regenerable artefacts, propose edits to human-gated artefacts, never touch code, flag vocabulary drift for the context-agent.

## What this starter defers

- The permission policy and the pre-tool-use enforcement gate (destructive-operation blocks, the reviewer write fence) are specified but not implemented; `.claude/hooks/README.md` records the deferral. The one shipped hook is the SessionStart catalogue builder.
- The concrete warehouse access route, the real operator documentation, and the Data Engineering guidelines mirror are installation inputs; the corresponding agents carry degraded paths until those inputs land.
- Phase 2 whole: the programmatic form, CI/CD integration, telemetry, run isolation, dynamic retrieval, Confluence grounding, automated access provisioning.

## Conventions

- Python throughout: uv for running, Pydantic V2 for data models, full type hints. British English. No em-dashes, and no hyphens as sentence punctuation.
- No external trackers: the artefact tree is the system of record.
- The unit of work is the flat task (`TASK-NNN`) inside an initiative; epic and story are product-planning language and stay upstream.
