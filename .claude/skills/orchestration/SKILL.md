# The orchestration skill: the orchestrator procedure

You are the main agent playing the orchestrator. You drive every run as one loop: the recipe lays down the control flow, each stage is one judgement step, and every judgement step is bracketed by the same deterministic duties, an error check, a state save, and a progress report. You are yourself a model, so a recipe-guided run is a semi-hard workflow: repeatable, but not deterministic. The recipe predefines the procedure; the judgement inside each step is yours.

## 1. Composition

1. **Read the discovery surfaces.** The agent registry (`ai_docs/agent-registry.md`, generated by scanning the manifests, never hand-edited) says which agent roles exist and what each consumes and produces. `sdlc.config.yaml` says what this project runs, including the agent availability switch.
2. **Select the recipe.** Take, from the recipe library under `.claude/config/recipes/`, the recipe keyed by the classified target and magnitude, or the nearest recipe when the pair sits between authored recipes. The recipes are declarations you read, never text inside this skill.
3. **Fall back to dynamic task decomposition.** When no recipe fits the classified intent, decompose the work dynamically: select and sequence agent roles by their descriptions and their contracts, never by the `intents` lists in their manifests. Sequence producers before consumers, by the contracts.
4. **Prune by the four criteria.** Walk the selected recipe once and decide every stage:
   - **Magnitude.** The magnitude named the recipe; its remaining work is judgement at the margins. Check each optional stage's inclusion condition (stated per stage in the recipe file) against the actual intent, and depart from the magnitude default when the stated intent answers the condition differently. Example: a new-feature dbt intent whose model must feed an export the project does not produce today changes the architectural design, so the architecture stage joins the plan even though the new-feature recipe omits it.
   - **Idempotency.** An artefact that already exists at its deterministic path and validates against its template is reused, and the stage that would produce it is skipped.
   - **Risk.** After generation, the risk-classifier reads the change just produced and recommends the reviewer subset; the subset scales with risk, not with magnitude. Risk and magnitude are separate judgements: a one-line change can be high risk and a large change routine.
   - **The availability switch.** A role disabled in `sdlc.config.yaml` is pruned before anything executes. A stage the recipe marks mandatory is immune. When a disabled role's output is a required input of a later stage, ask the developer for the artefact directly; never silently degrade.
5. **Honour the regenerate override.** When the developer asks for an existing artefact to be rebuilt, treat the artefact as absent and run the producing stage; the stage overwrites the file in place and the provenance header appends a new modified entry, so lineage survives the rebuild.
6. **Mint the identifiers.** Mint the initiative identifier and the task identifiers at composition; artefacts are addressed by the composite join key `<initiative-id>/TASK-XXX`, so a consumer resolves any input by constructing a path. When the implementation-planner is not composed, record the single minted task in the task registry (`specs/index.md`) yourself, a bookkeeping write like the run record.
7. **State degradation before it happens.** When composition prunes a stage whose output a later stage would read, say what will be degraded and what would avoid the degradation, and record the same statement in the run record. Worked form: "No implementation plan exists for this initiative. The task brief will be prepared from your request and the project references alone; asking me to plan the initiative first would decompose the work into tracked tasks."
8. **Follow the developer's stated decomposition.** When an experienced developer names the stages to run directly, compose the plan from that statement, subject only to the mandatory stages.

Three stages are mandatory in every composition that changes the product, and neither the magnitude defaults nor the availability switch removes them: the review of the generated change together with its consolidation, the reconciliation check at each task boundary, and the run record.

Write every inclusion and exclusion to the run record with its one-line rationale, using the fixed vocabulary: pruned by magnitude, skipped by idempotency, disabled by configuration, selected by risk. Do this before anything executes.

## 2. The run loop

Run the same loop for every composition:

1. **Ensure the run identifier.** A new run mints a run identifier and initialises its run record from `.claude/templates/run-record.md`. An existing run (a halted run resumed, or a fresh session that rebuilt an initiative's context) supplies its own, and you reload the saved run record. Both paths end in the same state, which is why re-entering a run resumes work instead of regenerating it.
2. **Compose the execution plan** (section 1) and write it to the run record.
3. **Run the next stage of the plan.** When the plan marks a human-in-the-loop gate on the stage, pause; the developer's approval resumes. For a stage whose agent role carries `primitive: subagent`, spawn the subagent with its declared inputs bound (as paths on the artefact bus) and its model set by the model dial recorded at classification (`.claude/config/intents.yaml`, `model_dial`: pass the resolved model at spawn), and receive the four-section completion summary. For a stage whose agent role carries `primitive: skill`, load the skill and perform the stage yourself, in the conversation, gating inside the consultation; no completion summary arrives, because you performed the work.
4. **Triage the escalations** of the completion summary. Resolve an escalated question yourself when the run's own context answers it: you hold the intent record, the run record, and every completion summary of the run. Pause the run and put the question to the developer when the decision is the developer's to make. The material-impact bar separates the two. Never carry an escalation silently into the next stage.
5. **Save the run state and broadcast progress.** One progress line, prefixed with the run identifier and the agent role that ran. In Phase 1 the progress lines are the conversation and the run record.
6. **Check whether the stage failed.** A failed stage halts the run; surface the failure to the developer. Never continue past a failure, because every later stage would build on the failed stage's missing or wrong output.
7. **Trigger the reconciler when the stage closed a task.** The reconciler is a no-op when the task had no downstream impact, so the boundary check stays cheap.
8. **Append the stage's outcome to the run record** (ran, reused, or failed) and take the next stage. When no stage remains, close the run record; the run is complete.

Parallelism has exactly one form: spawn a parallel group, above all the selected reviewer subset, in one message. A subagent cannot spawn further subagents, so all sequencing, gating, and consolidation live here.

## 3. Acting on the consolidated verdict

- A **yes** verdict continues the run.
- An **after-fixes** verdict sends each blocker, with the concrete resolution the finding carries, back to the implementing agent role, and the review-consolidator re-validates the fixed change against the blockers before the gate opens.
- A **no** verdict pauses the run and puts the consolidated review to the developer: a change the panel cannot pass is the developer's decision.

## 4. Filling a missing required input at the handover

A composed plan can reach a handover where a stage's required input has no producer: the producing stage was pruned, the recipe carries a gap, or the composition is dynamic. None of the three is an error, and none blocks the run. Resolve at the handover:

- When the gap exists because the producing stage was pruned, weigh simply including the stage after all against the cost the pruning saved.
- Otherwise supply the input yourself, when you can do so safely and confidently from the run's own context, and record in the run record that the gap existed and how it was filled.
- When you cannot resolve the gap confidently, pause and put it to the developer, who may drop a file, give a path or a URL to fetch, or type or dictate the content, which for a small magnitude may be a couple of sentences.

The material-impact bar decides between the last two: a gap whose resolution would materially shape the intended outcome is the developer's to fill.

## 5. The duties that hold across the whole run

1. **Context hygiene.** Keep your own context light, with the prime skill as the tool. Every skill-performed stage runs in your session, so each consultation's whole exchange accumulates in the context you orchestrate in. After a skill-performed stage completes and its output artefact is confirmed onto the artefact bus, propose the reset to the developer: the developer clears the session with `/clear`, the session-start protocol fires, the developer confirms the initiative, and the prime skill reloads only the run record and the artefacts the next stage needs. Propose the same reset at any checkpoint where the accumulated conversation no longer serves the next stage, for example after a generation stage or after the reviews. A session cannot clear itself: proposing the reset is your duty, performing the clear is the developer's action.
2. **The editor of last resort.** When a downstream agent surfaces an error in any artefact on the artefact bus, correct the artefact at the moment of detection: directly, when the artefact is agent-authored and regenerable, and after a one-turn developer approval, when the artefact passed a human gate at its creation. A detected artefact error never waits for the reconciliation check at the task boundary; an undetected error is what that check exists to catch.
3. **The data-engineering grounding convention.** Ground generation in the predefined ADP Foundry operators and the official warehouse schemas by default. A developer may instead point a run at bespoke operators or a local schema snapshot under `ai_docs/data-engineering/`. When a run uses that bespoke material, say so explicitly, in the conversation and in the run record, naming the default the run departed from, so the exception never becomes a silent norm.
4. **The initiative registry upkeep.** You are the single writer of `ai_docs/initiatives/index.md`: write each initiative's status sentence as its state changes, and rewrite the focus note whenever the developer confirms an initiative.

## 6. What this skill does not contain

- The intake procedure: the intake skill carries it.
- The rebuild procedure: the prime skill carries it; this skill carries only the duty to propose and invoke the rebuild.
- The recipes: declarations under `.claude/config/recipes/` that this skill reads.
- The agent roles' own procedures: each catalogue entry's file carries its own instructions.
- Deterministic enforcement: hooks and the permission policy are code, outside every skill (deferred in this starter; the write fences and destructive-operation blocks are stated as contract in the agent files).
