# Agentic SDLC

The Phase 1 starter of an agentic software development lifecycle harness for Claude Code. A team copies the harness into a project repository, and the repository's Claude Code becomes a lifecycle agent: the developer states a need in plain language, and the pipeline carries the need through requirements, architecture, planning, generation, testing, and review, with the developer approving at defined gates.

This starter accompanies the specification document "Agentic SDLC Specification and Architecture"; the specification is the authority for every design decision here. Hooks beyond the session-start catalogue, the permission policy, and the pre-tool-use gates are specified there and deliberately deferred in this starter (`.claude/hooks/README.md` records what is deferred).

## The shape of the system

- One agent. Claude Code is the only agent; everything in this repository is harness: instructions, agent definitions, skills, recipes, templates, scripts, and configuration that specialise it.
- The main agent plays two roles: the concierge, which classifies the developer's intent on target (eight kinds of work) and magnitude (bug fix up to new project), and the orchestrator, which composes a pruned execution plan from a recipe, spawns a subagent per stage or performs a skill stage in the conversation, enforces the gates, and writes the run record.
- Agents coordinate only through the artefact bus: a two-tier tree (default `ai_docs/`) holding a durable project spine plus one workspace per initiative. No agent obtains work by calling another.
- One pipeline converges through shared upstream stages, diverges at exactly one generation stage with three targets (ADP Foundry YAML, dbt model, ad-hoc code), and converges again through the shared downstream (risk-scoped review panel, consolidated verdict, walkthrough, reconciliation).
- Every agent role declares a machine-readable manifest in its frontmatter; `ai_docs/agent-registry.md` is generated by scanning the manifests, never hand-edited.

## Quick start

From this repository, copy the harness into a target repository:

```bash
./install.sh /path/to/target-repo
```

Inside the target repository, open Claude Code (restart the session so the agents and skills register) and run:

```txt
/setup
```

Setup writes `sdlc.config.yaml` (a greenfield interview, or a brownfield survey of the existing repository), scaffolds the shared memory, and generates the agent registry. Then either:

- **State an intent.** "The stg_orders model double-counts cancelled orders" classifies as a dbt-model bug fix and earns a short plan; "we need a new ingestion pipeline for the supplier feed" earns a deeper one. The orchestrator shows the composed plan and its rationale before anything runs.
- **Drive by hand.** Invoke one skill at a time (`/project-charter`, `/requirements-navigator`, `/arch-blueprint`, `/implementation-planner`, `/adhoc-code-implement-loop`, `/prime`), or ask the main agent to run any named subagent role (a branch review needs nothing upstream: ask for the review panel on a diff).

## Where things live

- `CLAUDE.md` is the thin constitution: the two roles, the always-on rules, the loading rule. Everything procedural loads on demand.
- `.claude/skills/` carries the harness-core skills (intake, orchestration, prime) and the skill-realised agent roles (the four gated consultations, the ad-hoc code implement loop, access provisioning, setup).
- `.claude/agents/core/` carries the reusable subagents, including the ten-dimension review panel; `.claude/agents/data-engineering/` carries the profile agents: two grounding retrievals (warehouse schema, ADP Foundry operator reference) and two generators (ADP Foundry YAML, dbt models).
- `.claude/config/` carries the intent vocabulary, the recipe library, the profile schema with worked examples, and the failure-pattern catalogue.
- `.claude/templates/` fixes the shape of every artefact type; every artefact carries an append-only provenance header.
- `docs/` explains the methodology and the guides; `DESIGN.md` is the design summary that binds this starter to the specification.
